上一篇聊了终端里的 OpenCode，但很多开发者的日常工作重心在 IDE 里。VS Code 侧边栏、内联补全、文件树——这些都是日常开发的高频触点。如果你在找一个 GitHub Copilot 的开源替代品，那 Cline 就是目前最值得关注的一个——63.1k GitHub Star，Apache 2.0 开源，同样是 IDE 插件形态，同样支持对话式编程和任务执行。

## 一、Cline 是什么

Cline 的前身叫 Claude Dev，最早是一个 VS Code 扩展。后来改名 Cline，逐步发展出 CLI、JetBrains 插件、看板多代理面板、SDK 等多种形态。

它和 GitHub Copilot 的关系值得说清楚。2026 年的 Copilot 已经不只是代码补全工具了——它有了 Agent 模式，可以对话、规划、执行多步任务。所以 Cline 和 Copilot 的核心差异不在于“一个是 Agent，一个是补全”，而在于：

- Copilot 是**商业产品**，有免费版和付费版，深度绑定 GitHub 生态；
- Cline 是**开源产品**，Apache 2.0，不绑定任何厂商和平台。

两者的使用体验也有区别：Cline 没有内联代码补全功能，只有对话式交互；Copilot 两者都有。如果你习惯“写一半让 AI 补全”的工作流，Cline 帮不上忙；但如果你想要一个开源可控、模型自由切换、可以本地部署的 IDE 编程助手，Cline 是目前最成熟的选择。

## 二、Cline 提供的使用形态

和 OpenCode 类似，Cline 也提供了多种使用形态，而且覆盖面更广：

**VS Code 扩展**（主力形态）。在 VS Code 扩展商店直接安装，左侧 Activity Bar 打开面板，用自然语言描述任务即可。这是大多数用户的第一接触点，也是对 IDE 用户最友好的形态。

**JetBrains 插件**。IntelliJ IDEA、PyCharm、GoLand 等 JetBrains 系 IDE 都可以安装。功能和 VS Code 版一致。

**CLI**。命令行版本，支持交互式使用和无头模式。无头模式可以嵌入 CI/CD 流水线，按 Cron 定时运行代理任务。

**看板多代理任务板**（Kanban）。通过 Web 界面并行管理多个代理任务，每个代理独立工作，状态持久化。适合需要多个任务并行推进的场景。

**SDK**。提供完整的编程 API，可以构建自定义 AI 代理、工具链、多代理团队协作系统。

这五种形态共享同一套核心引擎（TypeScript 实现），配置和规则在不同形态间保持一致。

## 三、动手：安装与配置

**安装 VS Code 扩展**：在 VS Code 扩展市场搜索 `Cline`，点安装。左侧 Activity Bar 会出现 Cline 图标，点进去就是对话面板。

**连接本地模型**。在 Cline 面板的设置里，选择 `Ollama` 作为 API Provider，Base URL 填 `http://localhost:11434`，Model 填你本地跑的模型名（如 `qwen2.5:14b`）。这样就能用内网的 AI 能力了，代码不出本地。

**配置项目规则**。在项目根目录创建 `.clinerules` 文件：

```text
# 编码规范
- 使用 TypeScript 严格模式
- 所有函数必须有 JSDoc 注释
- 测试覆盖率不低于 80%

# 安全规则
- 禁止修改 src/auth/ 目录下的文件，除非显式要求
- 数据库迁移脚本必须人工审查后才能执行

# 自动批准（谨慎使用）
- 允许自动批准：读取文件、运行 npm test
- 必须审批：git push、npm publish、rm -rf
```

Cline 会自动读取这个文件，所有操作都会遵守你定义的规则。

**第一次使用**：打开 Cline 面板，输入任务描述，选 Plan 模式让它先出计划，确认后切到 Act 模式执行。

## 四、Plan/Act 双模式

Cline 最核心的设计是 **Plan/Act 双模式**。

**Plan 模式**：代理只读不写。你描述一个任务，Cline 先探索代码库，理解项目结构，然后提出一个结构化的执行计划。它会告诉你：我打算改哪些文件、做哪些操作、预期结果是什么。如果你不满意，可以继续讨论，调整方向。

**Act 模式**：确认计划后，切换到 Act 模式，Cline 开始执行。每一个文件编辑、每一个终端命令，默认都需要你手动批准。你可以针对某些类型的操作开启自动批准（比如读取文件），但对危险操作（如 `git push`、`rm -rf`、数据库迁移）保持审批。

这个设计解决了一个关键问题：**执行能力和安全性之间的平衡。** 完全自动的 Agent 太危险——你不知道它会不会删了不该删的东西。但每一步都要确认又太慢——读个文件也要点一下确认，效率极低。Plan/Act 模式让用户在战略层面把关（Plan 确认），在执行层面放权（Act 自动批准安全操作），是目前比较务实的做法。

## 五、核心能力

**MCP 协议原生支持**。Cline 原生支持 MCP（Model Context Protocol），可以连接数据库、查询 API、管理云资源、获取内部文档——它不只是读你项目里的文件，还能访问组织层面的知识库。比如在执行数据库迁移前，自动查一下生产环境的表结构；在写 API 时，自动查阅公司内部的接口规范。

**无头浏览器测试**。这是 Cline 一个很特别的能力。大多数 AI 编程工具只能写前端代码，不能看效果。Cline 可以启动无头浏览器，截取页面截图，验证 UI 改动是否正确。你让它“把按钮颜色改成蓝色”，它会改了代码、刷新页面、截图确认按钮确实变蓝了。这个能力在前后端分离的项目里特别实用。

**BYOK 模型接入**。Cline 不绑定任何模型供应商。它支持 Anthropic、OpenAI、Google、Mistral、DeepSeek、OpenRouter（200+ 模型）、AWS Bedrock、Azure、Ollama 本地模型——几乎你能想到的模型提供商，它都支持。你可以先用便宜的模型做规划，再用高端模型执行复杂重构，灵活控制成本。

**`.clinerules` 项目规则**。在项目根目录下放一个 `.clinerules` 文件，定义编码规范、允许自动批准的命令、禁止修改的目录。团队共享同一套规则，不同开发者用 Cline 时行为一致。

**实时成本透明**。每一个操作都显示实时 Token 用量和预估费用。没有月费，成本完全由你的 API 使用量决定。

## 六、与 Copilot、Cursor 的对比

| 维度 | Cline | Cursor | GitHub Copilot |
|------|-------|--------|----------------|
| 开源 | Apache 2.0，完全开源 | 闭源 | 闭源 |
| 内联自动补全 | 无 | 有 | 有 |
| 工作方式 | 对话式，Plan/Act 双模式 | 对话式 + Composer 代理 | 对话式 + Agent 模式 + 内联补全 |
| 编辑器要求 | VS Code / JetBrains 标准扩展 | VS Code 独立分支 | 多种 IDE 扩展 |
| 商业模式 | 开源免费 + 自带 API Key | $20/月订阅 | 免费/Pro/Enterprise |
| 模型选择 | BYOK，支持几乎所有主流模型，包括本地模型 | 部分支持 BYOK | 内置多模型（Claude/GPT 等） |
| 审批门控 | 每步需审批（可配置自动批准） | 有类似机制 | 有安全控制 |
| MCP 支持 | 原生支持 | 有限 | 支持 |

一个关键差异：**Cline 没有内联代码补全功能。** 如果你习惯“我写一半、AI 补全另一半”的工作流，Cline 就不适合你——它只有对话式交互。但如果你想要一个开源可控、模型自由切换、可以本地部署的 Copilot 替代品，Cline 是最成熟的选择。

## 七、企业自建视角

从企业自建的角度，Cline 有几个实打实的优势：

**数据不出门**。BYOK 模式意味着你可以用自己的 API Key，甚至完全用本地模型（通过 Ollama），代码和对话完全在内网。

**零订阅费**。没有月费，成本完全按 API 用量计算。团队规模越大，相比订阅制方案（每人每月 $20-$39）的优势越明显。

**标准 VS Code 扩展**。不需要让团队换编辑器，不需要推一个“AI 原生 IDE”。Cline 只是一个扩展，安装即用。

**审批门控**。每步操作都需要人工批准，这是企业环境的基本安全保障。你可以通过 `.clinerules` 定义哪些操作可以自动批准、哪些必须审批。

**SDK 可编程**。如果你需要定制更深的集成，Cline 的 SDK 提供了完整的编程接口。你可以基于 Cline 的核心引擎构建自己的工具链、多代理协作系统。

## 八、小结

Cline 和上一篇的 OpenCode，解决的其实是同一类问题——用开源方案替代商业 AI 编程工具。但它们的切入路径不同：

- **OpenCode** 从终端切入，CLI/TUI 为主，LSP 原生集成是亮点，适合习惯在终端里工作的开发者。
- **Cline** 从 IDE 切入，VS Code 扩展为主，Plan/Act 双模式是亮点，适合习惯在 IDE 里工作的开发者。

两个都是开源、都支持本地模型、都不绑定厂商。你不需要二选一——在终端里写脚本时用 OpenCode，在 VS Code 里写业务代码时用 Cline，完全可以。

但它们有一个共同局限：**无论 OpenCode 还是 Cline，都只是“编程”阶段的 Agent。** 它们帮你写代码，但不能帮你自己跑测试、自己部署、自己修复环境问题。它们是在你的指令下行动的——你安排任务，它们执行。

有没有更进一步的工具——不只是“听指令写代码”，而是“领了任务自己做完”？这就是下一篇要聊的 OpenHands。

每天前进一小步，就是一个新的高度！
